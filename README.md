# MobX State Management Practice Project

Цей проєкт містить серію практичних задач для глибокого вивчення **MobX 6** та його інтеграції з **React**. Основна увага приділена роботі з реактивними масивами, механізмам автоматичного очищення стану (auto-cleanup) та роботі з асинхронними даними.

## 🛠 Технологічний стек

- **React** (Vite + TS)
- **MobX 6** (Observable, Actions, Computed)
- **MobX React Lite** (`observer`)
- **Axios** (для асинхронних запитів)
- **TypeScript** (Strong Typing)

---

## 📋 Реалізовані модулі

### 1. Emoji Reaction Store (`EmojiStore`)

Система відстеження активності емодзі з логікою анти-спаму.

- **Функціонал:** Додавання емодзі з таймстемпом та автоматичне видалення запису рівно через 3 секунди.
- **Spam Detection:** Статус `isSpamming` активується, якщо в масиві > 5 елементів.
- **Ключові концепти:** `@observable` масив, `setTimeout` всередині екшенів, `runInAction`.

### 2. Network Diagnostics Store (`InternetSpeedStore`)

Моніторинг стабільності з'єднання на основі останніх даних.

- **Функціонал:** Збереження історії пінгу за 10 секунд та розрахунок середнього значення.
- **Status Logic:** Автоматична зміна стану (`Excellent` / `Unstable` / `Bad`) залежно від результатів обчислень.
- **Ключові концепти:** Складні математичні розрахунки в `@computed`, візуалізація стану через динамічні стилі.

### 3. Smart Device Queue Store (`SmartLightStore`)

Імітація черги команд для IoT-пристрою.

- **Функціонал:** Обмежена черга (max 3 команди) з послідовною асинхронною обробкою.
- **UI Control:** Блокування кнопок керування, якщо пристрій зайнятий (`processing`) або черга переповнена.
- **Ключові концепти:** FIFO (First In, First Out) логіка, керування станом завантаження пристрою.

### 4. Async API Store (`CommentStore`)

Інтеграція зовнішнього сервісу для отримання даних.

- **Функціонал:** Завантаження списку коментарів через Axios з обробкою станів `isLoading` та помилок.
- **Data Flow:** Винесення запитів у незалежний сервіс `getComments.ts`.
- **Ключові концепти:** Повна типізація API-відповідей, `useEffect` для ініціалізації завантаження, асинхронні `Actions`.

## 🏗 Архітектурні принципи

У проєкті суворо дотримано правил MobX:

1. **No Destructuring:** Стори не деструктуруються в компонентах, щоб не розривати зв'язок MobX-проксі з об'єктом.
2. **Action Strict Mode:** Будь-яка зміна стану (навіть у `finally` або `setTimeout`) обгорнута в `runInAction`.
3. **Separation of Concerns:** Логіка запитів до мережі відокремлена від логіки керування станом.
4. **Type Safety:** Використання `import type` для інтерфейсів, щоб відповідати правилам `verbatimModuleSyntax`.
